# PyTorch memory bandwidth: broadcasting vs `.expand()` at 256K batch scale

**Both implicit broadcasting and explicit `.expand()` are functionally identical in PyTorch's memory model**—neither allocates new memory. The real performance differentiator lies in what happens *after* expansion: certain operations force materialization, and torch.compile handles both patterns equivalently through stride manipulation. For large recommendation models with **256K samples**, the compounding effect of unnecessary materializations across thousands of operations can consume 10+ GB of unnecessary memory and fragment the allocator.

The critical insight: memory bandwidth bottlenecks in attention computation persist regardless of batch size because attention kernels fundamentally stall on DRAM reads. FlashAttention reduces HBM accesses by **up to 9×** and should be the baseline optimization before micro-optimizing broadcast patterns.

---

## Inductor treats broadcasting and expand identically

PyTorch's torch.compile backend (Inductor) represents both implicit broadcasting and explicit `.expand()` through the same IR construct: **ExpandView**. This is a zero-copy view operation that sets `stride=0` for expanded dimensions, meaning the same memory location is accessed repeatedly without allocating new storage.

When Inductor generates Triton or C++ kernels, index expressions naturally handle stride=0 cases:

```python
# Generated Triton code handles strided access uniformly
tmp0 = tl.load(in_ptr + (x0 * stride0 + x1 * stride1), mask=xmask)
# When stride=0, multiple output elements map to same input element
```

The lowering process in `torch/_inductor/lowering.py` creates ExpandView nodes for both patterns, which share the underlying storage with modified stride metadata. **There is no dedicated "broadcast elimination pass"** in Inductor—broadcasting is handled implicitly through stride manipulation in the IR and index calculation in codegen.

A subtle difference emerges in **graph structure visibility**: explicit expand creates a visible FX graph node, while implicit broadcasting may be "inlined" into operator lowering. This can affect fusion boundaries in edge cases, though benchmarks show equivalent performance for typical patterns.

Different `torch.compile` modes (`reduce-overhead`, `max-autotune`) don't change broadcast handling logic—they affect kernel launch overhead and autotuning iteration, not the fundamental representation of expanded tensors.

## When `.expand()` triggers actual memory allocation

The zero-copy guarantee breaks under specific conditions. Understanding these triggers is critical for avoiding unexpected **multi-gigabyte allocations** at 256K batch scale.

**Operations that always force materialization:**

| Trigger | Memory impact at 256K × 1024 (fp32) |
|---------|-------------------------------------|
| `.contiguous()` on expanded tensor | **1.07 GB** full allocation |
| Device transfers (`.cuda()`, `.cpu()`) | Full tensor materialized on transfer |
| `.reshape()` when view impossible | Entire expanded tensor made contiguous |
| `.clone()` | Explicit copy of full size |
| `.view()` on expanded tensor | RuntimeError (incompatible strides) |

**Operations that can work with stride=0:**

- Element-wise ops with broadcasting (`torch.add`, `torch.mul`)
- Basic indexing (returns another view)
- `torch.matmul()` with broadcasting (though may materialize internally)
- Triton kernels generated by torch.compile (flexible pointer arithmetic)

**Critical footgun discovered**: Moving expanded tensors between devices triggers full materialization:

```python
n = int(1e8)
a = torch.randn(n)
a.cuda().expand((n, n))  # Works - expand AFTER moving
a.expand((n, n)).cuda()  # OOM! - 40+ GB materialized during transfer
```

**cuBLAS and cuDNN behavior** varies significantly. cuBLAS strided batched GEMM accepts stride parameters and can handle stride=0 for batch broadcasting. However, cuDNN convolutions and batch norm typically require contiguous inputs—non-contiguous tensors trigger `CUDNN_STATUS_NOT_SUPPORTED` errors. FlashAttention and scaled_dot_product_attention require `.stride()[-1] == 1` for all inputs.

## Memory bandwidth in attention remains the dominant bottleneck

For attention computation `softmax(QK^T/√d)V`, memory traffic—not compute—limits throughput regardless of batch size. Research shows attention kernels **do not benefit significantly from batching** because performance is constrained by DRAM reads.

**Standard attention memory traffic (naive implementation):**
- Total HBM accesses: O(N² + Nd) dominated by N×N intermediate matrices
- For N=4096, d=128 in fp32: **~331 MB per attention head** just in intermediate storage
- At batch=256K with N=128: attention scores alone would require **8.4 GB** if materialized

**FlashAttention transforms this equation:**

| Metric | Naive attention | FlashAttention |
|--------|-----------------|----------------|
| HBM accesses | O(N²) | O(N²d²/M) where M=SRAM |
| Memory complexity | O(N²) | O(N) |
| A100 throughput | ~25% theoretical | 50-73% theoretical |
| Training speedup | baseline | **7.6×** (GPT-2) |

FlashAttention achieves this through **tiling** (loading Q, K, V blocks into SRAM), **online softmax** (incremental computation without materializing N×N), and **recomputation** (not storing S, P matrices—recompute during backward).

**Broadcasting in attention masks** presents a specific tradeoff. A broadcast mask `[1, 1, N, N]` uses N² memory while `[B, H, N, N]` pre-expanded uses B×H×N² memory—potentially **2+ GB vs 8 KB** for typical dimensions. However, a critical PyTorch issue (#154363) revealed that GQA broadcasting without `enable_gqa=True` may dispatch to *less* memory-efficient backends, paradoxically increasing memory usage.

## Concrete memory footprint at 256K batch reveals identical baseline costs

For pattern comparison with `batch_size=262144` and `feature_dim=1024`:

```python
# Pattern A: explicit expand
base = torch.randn(1, 1024, device='cuda')  # 4 KB
expanded = base.expand(262144, -1)          # 0 bytes (view)
result = expanded + other                    # 1.07 GB allocated

# Pattern B: broadcasting
base = torch.randn(1, 1024, device='cuda')  # 4 KB
result = base + other                        # 1.07 GB allocated (identical)
```

**Both patterns allocate identical memory for the result tensor.** The difference emerges in edge cases where explicit expand introduces realization points earlier in the graph, potentially fragmenting the allocator or preventing fusion.

**Compounding effects across thousands of operations:**

The real cost manifests in aggregate. For 1000 layers each creating 1GB intermediates:
- **Best case** (immediate GC, buffer reuse): ~1-2 GB concurrent intermediates
- **Worst case** (accumulated references): 10+ GB with severe fragmentation

PyTorch's caching allocator tracks this through `inactive_split_bytes`—non-contiguous free blocks that can't satisfy large allocations despite total free memory appearing sufficient.

**Profiling recommendation:**

```python
torch.cuda.memory._record_memory_history(max_entries=100000)
# Run training iteration
torch.cuda.memory._dump_snapshot("snapshot.pickle")
# Visualize at pytorch.org/memory_viz
```

The memory timeline visualization reveals whether broadcasting or expand patterns cause fragmentation through the pattern of allocations and frees over time.

## Hoisting operations from conditionals requires autocast awareness

For recommendation models processing 256K samples through thousands of operations, hoisting common computations outside conditionals compounds small savings into meaningful optimization.

**Pattern transformation for unsqueeze/expand:**

```python
# Before: repeated computation
def forward(self, x, use_bias):
    if use_bias:
        bias = self.bias.unsqueeze(0)  # Computed every call
        return x + bias
    return x

# After: rely on broadcasting (no explicit unsqueeze needed)
def forward(self, x, use_bias):
    if use_bias:
        return x + self.bias  # Broadcasting handles shape mismatch
    return x
```

**Autocast-safe caching pattern:**

The challenge with hoisted tensors is dtype compatibility across autocast boundaries. Pre-computed float32 tensors inside autocast regions cause dtype mismatches.

```python
class AutocastSafeModule(nn.Module):
    def __init__(self):
        super().__init__()
        self.weight = nn.Parameter(torch.randn(1024, 1024))
        self._precomputed = {}

    def _get_cached(self, dtype, device):
        key = (dtype, device)
        if key not in self._precomputed:
            self._precomputed[key] = self.weight.to(dtype=dtype, device=device)
        return self._precomputed[key]

    def forward(self, x):
        # Automatically uses correct dtype inside/outside autocast
        w = self._get_cached(x.dtype, x.device)
        return x @ w
```

**Critical autocast rules:**
- Never call `.half()` or `.bfloat16()` manually inside autocast—it handles casting automatically
- In-place operations don't autocast (`a.addmm_()` won't work; use `a.addmm()`)
- Store buffers in float32 and let autocast convert on-the-fly

**Using torch.fx for systematic identification:**

```python
import torch.fx as fx

def find_hoistable_ops(model, example_input):
    traced = fx.symbolic_trace(model)
    hoistable = []
    for node in traced.graph.nodes:
        if node.target in [torch.unsqueeze, torch.expand]:
            # Check if inputs are constants/parameters
            hoistable.append(node)
    return hoistable
```

## Practical recommendations for 256K batch recommendation models

**Memory hierarchy priorities:**

1. **Use FlashAttention unconditionally** for sequences > 128—eliminates N² intermediate storage
2. **Prefer broadcasting over explicit expand** for cleaner graphs and equivalent compiler treatment
3. **Never transfer expanded tensors across devices**—expand after `.cuda()`, not before
4. **Fuse LayerNorm + Swish** using cuDNN 9.x or Apex kernels—reduces kernel launch overhead
5. **Profile with memory snapshots** before optimizing—locate actual fragmentation sources

**For SwishLayerNorm specifically:**

cuDNN 9.4+ supports LayerNorm/RMSNorm with activation fusion including Swish. The fusion eliminates intermediate tensor writes between the normalization and activation. TensorRT-LLM demonstrates this pattern: "combining LayerNorm, matmuls, activations into single CUDA kernels."

**Compiler-assisted optimization:**

```python
model = torch.compile(model, options={
    "epilogue_fusion": True,   # Fuses pointwise into templates
    "max_autotune": True,      # Profile matmul configs
})
```

Inductor's memory planning (`config.memory_planning`, `config.allow_buffer_reuse`) can automatically reuse buffers when lifetimes don't overlap, reducing peak memory without code changes.

## Conclusion

The broadcasting vs expand debate resolves to a draw at the IR level—both produce identical ExpandView representations with stride=0 and zero memory allocation. The real optimization surface lies in **avoiding materialization triggers** (device transfers, contiguous calls, incompatible reshapes) and **reducing memory traffic** through fused attention kernels. For 256K batch recommendation workloads, FlashAttention provides 7-9× memory traffic reduction that dwarfs any micro-optimization of broadcast patterns. Hoisting operations from conditionals delivers marginal per-operation savings that compound across thousands of layers, but requires careful autocast-aware caching to avoid dtype mismatches in mixed-precision training.
